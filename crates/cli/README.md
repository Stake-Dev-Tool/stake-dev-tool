# `sdt` — Stake Dev Tool CLI

Push a math folder to the Stake Dev Tool cloud as a new **revision** from a CI
pipeline. Math is generated by simulations, so the real workflow is "math CI
pushes a revision" — `sdt push` hashes the folder, uploads only the blobs the
server doesn't already have (content-addressed dedup), and commits the revision.

## Install

Build the `sdt` binary from the workspace:

```bash
cargo build -p cli --release
# binary at target/release/sdt (sdt.exe on Windows)
```

Or run it straight from the repo:

```bash
cargo run -p cli --release -- push ./math/my-game --workspace acme --game my-game -m "rev from CI"
```

## Configuration

Every command resolves the server URL and API token from three sources, in
**precedence order — flag > environment > config file**:

| Setting | Flag       | Environment  | Config key |
| ------- | ---------- | ------------ | ---------- |
| Server  | `--server` | `SDT_SERVER` | `server`   |
| Token   | `--token`  | `SDT_TOKEN`  | `token`    |

The server defaults to `http://127.0.0.1:8080` when nothing is set. The config
file lives at `~/.config/sdt/config.toml` and is written by `sdt login --save`
(mode `600` on Unix):

```toml
server = "https://cloud.example.com"
token  = "sdt_pat_…"
```

## Login (device flow)

Interactively mint a token with the `push:math` scope:

```bash
sdt login          # prints the token to stdout, an export hint to stderr
sdt login --save   # also writes it to ~/.config/sdt/config.toml
```

`login` requests a device code, prints a verification URL + user code, then
polls until you approve it in the browser. The token is printed **once** to
stdout so it can be captured (`TOKEN=$(sdt login)`); everything else goes to
stderr.

## Push

```bash
sdt push <PATH> --workspace <slug> --game <slug> -m "<message>" [options]
```

The `<PATH>` must be a math folder containing `index.json`. Every regular file
is included recursively (symlinks are rejected, dotfiles skipped); paths are
sent relative with forward slashes.

| Flag              | Purpose                                                        |
| ----------------- | ------------------------------------------------------------- |
| `--workspace`     | Workspace slug (required).                                     |
| `--game`          | Game slug (required).                                          |
| `-m`, `--message` | Commit message (required).                                     |
| `--parent <n>`    | Parent revision number (defaults to the server's latest).     |
| `--wait-stats`    | After committing, poll for bet-stats and print a table.       |
| `--timeout <s>`   | Timeout for `--wait-stats` (default 120).                     |
| `--no-progress`   | Plain log lines instead of progress bars (auto off non-TTY).  |
| `--json`          | Print the raw revision JSON to stdout (for CI scripting).     |

What it does:

1. Scans + validates the folder, streams a SHA-256 over each file, prints a
   summary (file count, total size, largest file).
2. Asks the server which blob hashes it's missing and uploads **only those**,
   streaming each file from disk (a book is never buffered whole in memory),
   with per-file progress and up to 3 retries (1s/2s/4s backoff).
3. Commits the revision. On a `missing_blobs` race it re-uploads the listed
   hashes and retries the commit once; on a `stale_parent` conflict it aborts
   with a clear message.
4. Prints the new revision number to **stdout** and a recap (uploaded X of Y
   files, bytes sent, bytes deduplicated) to stderr.

Output streams: human text and progress go to **stderr**; the machine-usable
result (the revision number, or the full JSON under `--json`) goes to
**stdout**.

### Exit codes

| Code | Meaning                                             |
| ---- | --------------------------------------------------- |
| `0`  | Success.                                            |
| `1`  | Usage / validation (e.g. not a math folder).        |
| `2`  | Authentication (401/403 — bad token or missing scope). |
| `3`  | Server or network failure.                          |

## Revisions

List a game's revisions as a table (stderr), or raw JSON (stdout) with `--json`:

```bash
sdt revisions --workspace acme --game my-game --limit 20
sdt revisions --workspace acme --game my-game --json
```

## CI usage (GitHub Actions)

Store a `push:math`-scoped token as the repository secret `SDT_TOKEN`, then:

```yaml
name: Push math revision
on:
  push:
    branches: [main]

jobs:
  push-math:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # …generate the math folder (simulations) into ./math/my-game…

      - name: Install sdt
        run: cargo install --path crates/cli   # or download a release binary

      - name: Push revision
        env:
          SDT_SERVER: https://cloud.example.com
          SDT_TOKEN: ${{ secrets.SDT_TOKEN }}
        run: |
          sdt push ./math/my-game \
            --workspace acme \
            --game my-game \
            --message "math: ${{ github.sha }}" \
            --no-progress \
            --json > revision.json
          echo "Pushed revision $(jq .number revision.json)"
```

`SDT_TOKEN` and `SDT_SERVER` are picked up from the environment, so no flags are
needed. `--no-progress` keeps CI logs clean (it's also auto-disabled when the
output isn't a TTY), and `--json` gives a machine-readable result on stdout.
